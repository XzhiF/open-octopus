# 需求澄清方法论

## 触发条件
当 swarm 节点进行需求澄清头脑风暴时加载此 skill。

## 澄清流程

### 第一步：功能完整性分析
无论什么 idea，都要思考功能完整性和所有环节的闭环：
- 输入端：用户从哪来？什么角色？
- 处理端：核心业务逻辑是什么？
- 输出端：结果呈现给谁？以什么形式？
- 异常端：失败、超时、并发怎么处理？
- 边界端：数据量极限、并发极限、权限极限

### 第一步半：UI Wireframe（当 Idea 涉及界面时）

如果 Idea 涉及用户界面（页面、表单、弹窗、导航变更等），在生成问题清单前先产出 ASCII wireframe，作为澄清的一部分让用户确认。

**Wireframe 格式：**

```
┌─────────────────────────────────────────┐
│  Logo    [搜索...]     [头像 ▾]         │
├──────────┬──────────────────────────────┤
│          │                              │
│ 侧边栏   │    主内容区                   │
│          │                              │
│ ▸ 首页   │  ┌─────────────────────┐     │
│ ▸ 项目   │  │  Dashboard Cards    │     │
│ ▸ 设置   │  └─────────────────────┘     │
│          │                              │
└──────────┴──────────────────────────────┘
```

**规则：**
- 用 `┌─┐│└─┘├┤┬┴┼` 画框线，`[按钮]` 表示可交互元素
- 标注关键交互：`点击 → 跳转`、`拖拽 → 排序`
- 每个主要页面/状态一个 wireframe（正常态 + 空态 + 错误态）
- wireframe 写入 questions.md 的「待澄清」区，用户确认后移入「已确认」
- 不涉及 UI 的 Idea 跳过此步骤

### 第二步：澄清问题清单生成
输出到 `.octopus/xzf/{feature}/02-clarification/questions.md`

#### 文件结构（三个区域）

文件分三个区域:「待澄清」、「用户补充」和「已确认」。每个问题只出现一次。

```markdown
# 澄清问题清单 — Round {N}

## 待澄清

### ❓ Dx: [问题标题] [优先级]
**背景**: [为什么需要澄清]
- **方案 A**: [名称] — [描述]
- **方案 B**: [名称] — [描述]
**⭐ 推荐 A**: A [适合原因]；B [不选原因]

## 用户补充

### 📝 Ux: 用户原话摘要
→ 分析 + 是否影响已确认项

## 已确认

### ✅ Dx: [问题标题]
→ 选项 + 一行用户回答
```

#### Round 1（文件不存在时）
- 分析问题，将新问题写入「待澄清」
- 每个问题必须包含 2-3 个方案，标注推荐项

**✅ 正确示例（A/B/C 选项 + 推荐标注）：**

```markdown
### ❓ D1: 数据库选型 [P0]
**背景**: 项目需要存储用户配置和工作流执行记录，需确定存储方案。
- **方案 A**: SQLite — 嵌入式数据库，零配置，单文件部署。无外部依赖，适合中小规模；并发写入受限
- **方案 B**: PostgreSQL — 成熟关系型数据库，并发能力强；需独立部署运维
- **方案 C**: Turso (libSQL) — SQLite 兼容边缘数据库，支持分布式；额外网络延迟
**⭐ 推荐 A**: A 零配置最契合单机部署目标；B 并发强但引入运维负担，当前规模不需要；C 分布式能力当前用不上且有供应商锁定风险

### ❓ D2: 认证方式 [P1]
**背景**: 需要保护 API 接口，防止未授权访问。
- **方案 A**: API Key + Header — 请求头携带 Key，服务端校验。实现简单无状态；需 HTTPS 防泄露
- **方案 B**: JWT Token — 签发短期 JWT 定期刷新。支持细粒度权限；需维护 refresh 逻辑
**⭐ 推荐 A**: A 实现成本最低，内部工具足够安全；B 权限粒度优势在当前场景无实际价值，复杂度不值得
```

**❌ 错误示例（只有推荐，缺少备选方案）：**

```markdown
### ❓ D1: 数据库选型 [P0]
**背景**: 项目需要存储用户配置。
**方案**: SQLite — 嵌入式数据库，零配置
```

#### Round 2+（文件已存在时）
1. 读取现有文件
2. 用户回答了的问题 → 移到「已确认」，格式: `### ✅ Dx: 标题\n→ 选项 + 一行用户回答`
3. 用户未回答的问题 → 保留在「待澄清」不动
4. 用户在评论中补充的新问题/约束 → 记录到「用户补充」区，格式: `### 📝 Ux: 用户原话摘要\n→ 分析 + 是否影响已确认项`
5. 上一轮「用户补充」已处理完毕的 → 移到「已确认」
6. 需要新追问的 → 追加到「待澄清」
7. 不重复已有问题，不重写整个文件

#### Synthesis 输出

Synthesis 是 approval 节点展示给用户的内容，**必须让用户看到完整选项才能做决策**。

读取刚写入的 questions.md，严格按以下格式输出：

##### 「待澄清」— 必须保留所有方案选项（不可压缩！）

每个问题必须列出**所有方案 A/B/C**，末尾用推荐行逐一点评并说明推荐理由。格式：

```
#### ❓ Dx: [标题] [优先级]
- **方案 A**: [名称] — [描述 + 优点/缺点]
- **方案 B**: [名称] — [描述 + 优点/缺点]
- **方案 C**: [名称]（如有）— [描述 + 优点/缺点]
**⭐ 推荐 A**: A [为什么适合]；B [为什么不适合当前场景]；C [取舍点]
```

**禁止**只输出推荐方案而省略备选方案。用户需要看到所有选项才能做出知情决策。

##### 「用户补充」— 原样输出

如有用户补充内容，完整输出。

##### 「已确认摘要」— 始终输出（一行摘要即可）

即使「待澄清」为空也要给用户审阅机会：
- ✅ **已确认决策概览**：每个 Dx 一行摘要，格式 `Dx: [标题] → [用户选择/结论一句话]`
- 🔍 **Research 项**：如有 R-x 项，列出标题和状态
- 📐 **UI Wireframe**：如有，简要描述布局要点

已确认项用一行摘要即可（决策已定，不需重复选项详情）。
完整内容已写入 questions.md，用户可查阅。

#### checklist_status 判定
「待澄清」+「用户补充」都已清空 → `"COMPLETE"`
有未处理项 → `"INCOMPLETE: {缺失项}"`

#### 阶段衔接（checklist COMPLETE 时）

当 checklist 判定为 COMPLETE，本轮 Synthesis 之后追加输出**初始验证问题**，供下游 verification-approval Round 1 直接展示给用户：

- 从「已确认」区的每个 Dx 决策推导验收标准和 E2E 场景
- 输出格式参照 octo-xzf-verification skill 的 Synthesis（推荐验证方案）：
  - 🏗️ **测试环境**：启动命令、端口、DB、测试工具
  - ✅ **验收条件概览**：每个 Dx 一行
  - 🎬 **E2E 场景列表**：每个 Sx 一行
  - 📋 **单元测试模块**：列出覆盖的模块名

这不是完整的验证规划（那是下一阶段的工作），只是首轮验证问题让用户尽早审阅。

## Research 项（需要专家深入理解的内容）

### R-1: Codebase Research
- **范围**: {哪些模块/文件需要专家重点研究}
- **目的**: {为什么需要理解这部分}
- **用户补充**: {用户可以提供的上下文/文档链接}

### R-2: 技术调研
- **范围**: {需要调研的技术方案/第三方库}
- **目的**: {为什么需要调研}
- **用户补充**: {已知的参考资料/对比候选}
```

> Research 项与功能澄清项、环境澄清项并列，不是独立阶段。用户在 approval 中一并回答。
> Research 答案下游流入 `technical-guide.md`（技术调研结论）和对应 spec 文件（codebase 研究结论）。

### 第三步：充分性评估
每轮澄清后评估：
- 是否所有核心功能都有对应澄清项
- 是否可以结束澄清进入下一阶段

checklist_status 判定:
- 「待澄清」+「用户补充」都已清空 → `"COMPLETE"`
- 有未处理项 → `"INCOMPLETE: {缺失项逗号分隔}"`

## 输出规范
- 每个澄清问题用 **方案 A / B / C** 格式列出 2-3 种方案，末尾用 **⭐ 推荐 X** 标注推荐项
- **Synthesis 输出中「待澄清」部分必须列出所有方案（A + B + C），不可只输出推荐方案**
- 用户在下一轮 approval 中回答或追加修改
- 问题清单是追加模式，每轮标记 Round N
