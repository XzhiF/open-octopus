# 验证策略规划

## 触发条件
当 verification-clarify 节点规划验证策略时加载此 skill。

## 核心理念

**推导，不是提问。** 验证策略是 agent 从 idea + research + 已确认需求主动推导出来的，不是向用户收集的。用户在 approval 中只做补充/修正，不作为主要信息来源。

## 验证策略流程

### 第一步：决策→验收条件映射

读取 `02-clarification/questions.md` 中「已确认」区的每个 Dx 决策，逐一推导：

- **通过标准**: 什么行为/结果证明这个决策正确实现了
- **反假跑条件**: 什么情况看起来通过但实际没测到（mock 永远返回 true、断言太宽泛、只测 happy path）

格式：

```markdown
### ✅ Dx: [决策标题]
- **通过标准**: [具体可观测的行为]
- **反假跑**: [可能假通过的场景 + 对应防护]
```

### 第二步：E2E 测试场景设计

从用户视角设计完整测试场景。每个场景包含：

```markdown
### 🎬 Sx: [场景名]
- **前置条件**: [需要什么初始状态]
- **操作步骤**:
  1. [具体操作 — 精确到点击/输入/导航]
  2. ...
- **预期结果**: [可断言的具体结果 — 不是"正常工作"]
- **边界/异常**: [相关的异常场景]
```

场景来源：
- P0 闭环中的每个功能点至少一个 happy path 场景
- 每个关键操作至少一个异常场景（空值、超时、权限不足）
- 用户流程的起点和终点必须覆盖

### 第三步：单元测试覆盖

从代码架构层面列出需要覆盖的逻辑路径：

```markdown
| 模块 | 文件 | 覆盖目标 | 关键 case |
|------|------|---------|----------|
| [模块名] | [文件路径] | [逻辑路径] | [具体 case 描述] |
```

覆盖原则：
- 数据校验（输入合法/非法/边界）
- 业务逻辑分支（成功/失败/异常处理）
- 状态流转（初始→中间→终态）

### 第四步：测试环境

从 workspace-topology.md 和 research 中提取已知环境信息，**仅记录不提问**：

```markdown
- **启动**: [命令 + 端口]
- **DB**: [类型 + 测试隔离方式]
- **E2E 工具**: [playwright/curl/等]
- **截图路径**: [目录]
- **测试数据**: [fixture/seed 方式]
```

仅当信息确实缺失且影响测试执行时，才作为待澄清项提出。

### 敏感凭据强制澄清

如验证策略涉及以下内容，**必须**生成问题让用户确认:

- 登录账号/密码、API Key、Token
- 测试环境中间件连接信息
- 特定数据集（种子数据、fixture、mock 数据源）

格式: `### ❓ Vx: [凭据] 标题 [CRITICAL]`

这类问题优先级始终为 CRITICAL，agent 无法自行推断。

## 输出文件

输出到 `.scratch/{feature}/02-clarification/verification.md`

### 文件结构

```markdown
# 验证策略 — Round {N}

## 待确认
（agent 推导但需要用户确认的项 — 如凭据、环境缺失信息）

## 用户补充
（用户在 approval comment 中的补充/修正）

## 已确认

### 验收条件
（第一步输出 — 每个 Dx 的通过标准 + 反假跑）

### E2E 测试场景
（第二步输出 — 完整场景列表）

### 单元测试覆盖
（第三步输出 — 模块/文件/case 表）

### 测试环境
（第四步输出 — 已知环境信息）
```

### Round 流程

#### Round 1
- 读取 idea.md + research + questions.md（已确认决策）
- 按四步推导验证策略
- 写入 verification.md
- 仅在凭据缺失或环境信息不足时产生「待确认」项

#### Round 2+
1. 读取现有 verification.md
2. 用户在 approval comment 中的补充/修正 → 记录到「用户补充」
3. 处理「用户补充」→ 更新策略 → 移到「已确认」
4. 需要追问的 → 追加到「待确认」

#### Synthesis 输出
读取刚写入的文件，输出以下三部分：

1. 「待确认」部分原样输出（需要用户提供的凭据、缺失信息等）
2. 「用户补充」部分原样输出
3. **「推荐验证方案」— 始终输出**，这是 agent 推荐的测试策略，用户可审阅、纠正或批准：
   - 🏗️ **测试环境**：启动命令、端口、DB、测试工具等完整列出（端口/命令错误会杀运行中服务，必须让用户审阅）
   - ✅ **验收条件概览**：每个 Dx 一行摘要，格式 `Dx: [标题] — [通过标准的一句话核心]`
   - 🎬 **E2E 场景列表**：每个 Sx 一行，格式 `Sx: [场景名]`
   - 📋 **单元测试模块**：列出覆盖的模块名

这是 agent 的推荐方案，不是最终确认。用户在 approval comment 中可纠正任何一项。
不要输出完整的验收条件详情和 E2E 步骤（太长），只输出概览。
完整内容已写入 verification.md，用户可查阅。

#### env_checklist_status 判定
- 「待确认」+「用户补充」都已清空 → `"COMPLETE"`
- 有未处理项 → `"INCOMPLETE: {缺失项}"`

## 输出规范
- 验收条件必须具体到可观测行为，不用"正常工作""功能正确"等模糊描述
- E2E 场景的操作步骤精确到 UI 元素（按钮名/输入框/菜单项），不写"点击相关按钮"
- 反假跑条件是每个验收条件的必备字段
- 测试场景数量 ≥ P0 功能决策数量
