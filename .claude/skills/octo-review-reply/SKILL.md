# 审查回复 — 辩证分析与文档更正

## 概述

当用户提交一份审查报告（由 `octo-design-vs-plan-review` 或类似流程产出），要求对报告结论进行分析和回应时，使用此技能。核心目标是：**专业辩证、独立判断、更正文档**。

这不是对审查意见的"附和"或"反驳"，而是基于项目实际约束和技术判断，对每个审查结论做出独立评估。正确的采纳，不正确的说明理由拒绝，部分正确的选取最合适的调整方案。

**输出目录（固定）：** `docs/superpowers/discussion/`
**回复命名（固定）：** `YYYY-MM-DD-<topic>-review-reply.md`

**开始时宣布：** "正在对审查报告进行辩证分析。"

## 流程

### 阶段 1：读取审查报告和相关文档

完整阅读审查报告，同时阅读被审查的设计文档和计划文档，建立完整上下文。必要时读取引用的源码文件验证审查结论的准确性。

**关键**：不要默认审查结论是正确的。审查者可能对项目约束、技术选型背景、实际使用场景有误解。用自己的判断重新评估每一项。

### 阶段 2：逐项辩证分析

对审查报告中的每个问题，按以下框架分析：

#### 分析框架

| 判定 | 定义 | 条件 |
|------|------|------|
| **✅ 采纳** | 审查结论正确，建议合理且可行 | 结论有证据支撑，建议与项目约束一致，修正后整体质量提升 |
| **⚠️ 调整** | 审查结论正确，但建议不合理或不最优 | 结论正确但建议忽略了项目约束、有更优替代方案、或修正代价与收益不匹配 |
| **❌ 拒绝** | 审查结论不正确或不值得修正 | 结论基于误解、实际影响极低、修正引入的风险大于收益 |

#### 判定原则

1. **结论验证**：审查说"X 是问题"——X 真的是问题吗？用源码和项目上下文验证
2. **建议评估**：审查建议"做 Y"——Y 在本项目可行吗？是否有更合适的方案？
3. **代价权衡**：修正此问题的代价（改动范围、风险、延迟）vs 收益（质量提升、风险降低）
4. **项目约束**：考虑项目技术栈、规模、阶段（开发期 vs 生产期）、团队偏好
5. **YAGNI**：不修正当前不需要的问题。注解"未来关注"而非立即行动

#### 拒绝理由模板

拒绝必须给出具体理由，而非笼统否定：

- **实际影响极低**：量化影响规模（文件数、调用频率、用户数）
- **修正引入新风险**：说明新风险是什么
- **过早优化**：当前规模下不需要此优化，未来规模增长时再处理
- **建议方案脆弱**：具体指出建议方案的弱点（如正则无法处理 YAML 引号字符串）
- **成本收益不匹配**：修正代价 > 收益的具体对比

### 阶段 3：写出回复文档

将分析结果写入 `docs/superpowers/discussion/YYYY-MM-DD-<topic>-review-reply.md`。

#### 回复格式

```markdown
# <主题> — Review 回复

> 日期: YYYY-MM-DD
> 回复对象: 审查报告路径

---

## 总体评价

N 个问题中：**X 个完全正确需修正**，**Y 个部分正确但建议需调整**，**Z 个不采纳**。

---

## 逐项分析

### 问题 1 (严重度)：标题 — ✅ 采纳 / ⚠️ 调整 / ❌ 拒绝

**结论**：[对审查结论的判断——正确/部分正确/不正确]

**[采纳时] 修正动作**：具体说明对设计/计划文档的修改内容

**[调整时] 修正方案**：说明为什么原建议不合适，给出替代方案及理由

**[拒绝时] 不采纳理由**：逐条列出拒绝理由（用上述模板）

---

## 修正汇总

| 项目 | 修正动作 |
|------|---------|
| 问题 1 | [具体修改内容] |
| ... | ... |
```

### 阶段 4：更正原始文档

根据回复文档中"修正汇总"表的每项，**实际修改**设计文档和计划文档：

1. 使用 Edit 工具修改设计文档中的相关段落
2. 使用 Edit 工具修改计划文档中的相关步骤
3. 每项修正应精确——只改需要改的部分，不做额外重构

**更正原则**：
- 只更正采纳和调整项，不更正拒绝项
- 保持文档其他部分不变
- 确保更正后的设计与计划之间不再有该矛盾
- 如果更正引入新的依赖关系（如新增共享类型文件），在计划中插入新任务

### 阶段 5：提交更正后的文档供用户审核

完成所有更正后，告知用户：

> "设计文档和计划文档已根据审查回复更正。请审核更新后的版本。"

不自动 commit。等待用户确认后才 commit。

## 关键原则

### 辩证而非附和（CRITICAL）

- 审查者可能对项目上下文不够了解——他们的"缺陷"可能是刻意的设计选择
- 审查者可能低估实际影响规模——他们建议的"优化"可能是不必要的
- 审查者可能提出理想方案但忽略了项目约束——需要找到最适合本项目的方案
- 不要为了"显得认真"而采纳所有意见——合理的拒绝比盲目的采纳更有价值

### 精确更正（CRITICAL）

- 更正必须精确指向需要修改的段落/步骤
- 用 Edit 工具而非 Write 重写——保持文档其他部分完整
- 每项更正独立可验证——改了什么、为什么改、改后效果

### 不过度更正

- 只更正审查报告中确认的问题
- 不顺手"改进"文档的其他部分（除非更正本身必然涉及）
- 不重构文档结构（除非重复定义问题要求提取共享文件）

### 保持设计-计划一致性

更正后的设计文档和计划文档必须保持一致：
- 设计描述的返回格式 → 计划代码的返回类型必须匹配
- 设计的路径计算逻辑 → 计划代码的实现必须匹配
- 设计新增的共享文件 → 计划必须有对应的创建任务

## 反模式

- 逐条附和审查意见，没有独立判断
- 拒绝但不给理由
- 调整但新建议比原建议更差
- 更正文档时改了不该改的部分（扩大变更范围）
- 只改设计不改计划（或只改计划不改设计），导致新的不一致
- 过度重构——审查说"提取共享类型"就顺便重构整个类型体系

## 与其他技能的关系

此技能是 `octo-design-vs-plan-review` 的下游：
- `octo-design-vs-plan-review` → 输出审查报告
- `octo-review-reply` → 输入审查报告，输出回复 + 更正后的文档


## 执行交接

更正后的文档，提供执行选项：

**"计划已更正并保存到 `docs/superpowers/plans/<filename>.md`。两种执行方式：**

**1. 子代理驱动（推荐）** - 每个任务调度一个新的子代理，任务间进行审查，快速迭代

**2. 内联执行** - 在当前会话中使用 executing-plans 执行任务，批量执行并设有检查点

**选哪种方式？"**

**如果选择子代理驱动：**
- **必需子技能：** 使用 superpowers:subagent-driven-development
- 每个任务一个新子代理 + 两阶段审查

**如果选择内联执行：**
- **必需子技能：** 使用 superpowers:executing-plans
- 批量执行并设有检查点供审查
