# Skill: octo-bug-investigation — 系统性排查 BUG 方法论

## 何时触发

在以下场景中使用此技能：

- 功能行为不符合预期，且原因不明
- 涉及前后端交互的复杂 BUG（保存、同步、状态更新等）
- 反复修复仍无法根治的问题
- 需要向团队解释 BUG 根因和修复方案时

## 六步排查法

### Step 1: 链路分层 — 先画地图再排查

**核心原则**: 在修改代码之前，必须先画出完整的数据流链路图。

**操作步骤**:
1. 从用户操作出发，画出触发 BUG 的完整链路
2. 标记每个节点（前端组件 → API 调用 → 后端路由 → 服务 → 数据库/文件）
3. 在每个节点标注：输入是什么、输出是什么、中间状态是什么

**示例**（文件保存冲突问题）：
```
用户点击保存
  → 前端 handleSave → saveFileEntry (api-client.ts)
    → 后端 PUT handler (file-routes.ts) → FileService.saveFile
      → 后端 handleChange (yjs.ts) 通过 chokidar 监听磁盘
        → WebSocket 同步到前端 Y.Doc
          → 前端 TextEditorTab 的 ytext.observe
```

**检查清单**:
- [ ] 是否画出了从用户操作到数据落地的完整链路？
- [ ] 每个节点是否标注了输入/输出？
- [ ] 是否遗漏了中间件、监听器、副作用等隐性节点？

---

### Step 2: 假设-验证 — 用代码和日志替代猜测

**核心原则**: 每个假设都必须有代码或日志支撑，不能靠"感觉"。

**操作步骤**:
1. 列出所有可能导致 BUG 的假设（一般 2-4 个）
2. 为每个假设设计验证方式：
   - 读源码确认逻辑
   - 添加 console.log 或断点
   - 查看网络请求/响应
   - 查看服务端日志
3. 逐一验证，排除不可能的假设
4. 对剩余假设进行优先级排序

**示例**: 保存文件冲突问题

| 假设 | 验证方式 | 结果 |
|------|---------|------|
| 后端 PUT 检查逻辑有误 | 读 file-routes.ts 第 63 行 | **确认** — `ytext !== diskContent` 永远 true |
| 前端 origin 判断有问题 | 读 y-monaco 源码确认 MonacoBinding 使用 `this` 作为 origin | 部分相关 |
| 网络请求没有正确发送 | 检查 Network tab 确认请求/响应 | 请求正常发送，返回 409 |

**检查清单**:
- [ ] 是否列出了所有可能的假设？
- [ ] 每个假设是否有代码/日志支撑？
- [ ] 是否排除了不可能的假设？

---

### Step 3: 边界条件分析 — 问自己"什么情况下会出问题"

**核心原则**: 在代码层面验证所有可能的输入组合。

**操作步骤**:
1. 找到 BUG 相关的核心条件判断
2. 列出所有可能的输入组合（正常输入 + 边界输入）
3. 逐个验证每种组合下代码的行为
4. 特别关注：空值、空字符串、极大/极小值、并发情况

**示例**:
```typescript
// 问题代码
const diskContent = fs.readFile(body.path).content
if (ytext.toString() !== diskContent) {
  return c.json({ conflict: true }, 409)
}
```

| 场景 | Y.Text 值 | 磁盘值 | 结果 | 是否合理 |
|------|----------|--------|------|---------|
| 用户编辑后保存 | "新内容" | "旧内容" | 409 | ❌ 应该正常保存 |
| 外部修改后保存 | "旧内容" | "新内容" | 409 | ✅ 正确 |
| 内容没变就保存 | "旧内容" | "旧内容" | 200 | ✅ 正确 |

**检查清单**:
- [ ] 是否覆盖了所有边界条件？
- [ ] 是否检查了空值/空字符串/特殊字符？
- [ ] 是否考虑了并发/时序问题？

---

### Step 4: 最小复现 — 剥离业务代码，验证核心逻辑

**核心原则**: 用最简单的代码验证假设，而不是在复杂业务逻辑中调试。

**操作步骤**:
1. 从业务代码中提取核心逻辑
2. 创建一个独立的测试文件或脚本
3. 用最少的输入复现问题
4. 验证修复方案是否有效

**示例**（React 18 批处理导致 spinner 不显示）:

```tsx
// 问题代码（React 18 自动批处理）
onClick={async () => {
  setRefreshing(true)      // ① 排队
  await onRefresh()         // ② React 不在此处 flush，继续执行
  setRefreshing(false)      // ③ 排队
}}                            // ①②③合并成一次更新：false → false

// 修复后（使用 flushSync 强制立即渲染）
import { flushSync } from "react-dom"
onClick={async () => {
  flushSync(() => setRefreshing(true))  // 强制立即渲染
  await onRefresh()
  setRefreshing(false)
}}
```

**检查清单**:
- [ ] 是否用最少的代码复现了问题？
- [ ] 是否去除了业务逻辑的干扰？
- [ ] 复现脚本是否可以独立运行？

---

### Step 5: 根因链追溯 — 不停于表面，问"为什么会这样"

**核心原则**: 找到最深层的根因，而不是停留在表面症状。

**操作步骤**:
1. 写出表面问题和直接原因
2. 连续追问 "为什么"（一般 3-5 次）
3. 直到找到可以修改的代码层面的根因

**示例**:

| 层级 | 问题 |
|------|------|
| 表面问题 | 一保存就冲突 |
| 直接原因 | 后端 PUT 返回 409 |
| 根因 | PUT handler 的比较逻辑错误（比较当前编辑内容 vs 磁盘内容） |
| 深层原因 | 缺少"原始内容"的基准点 |

| 层级 | 问题 |
|------|------|
| 表面问题 | 解决冲突后没保存到本地 |
| 直接原因 | `handleSaveMerged` 调用 save API 但返回 409 |
| 根因 | 后端 PUT handler 对所有非幂等保存都返回 409 |
| 深层原因 | 没有区分"正常保存"和"用户已确认解决冲突后的强制保存" |

**检查清单**:
- [ ] 是否追问到了代码层面的根因？
- [ ] 根因是否可以被修复？
- [ ] 修复根因是否会引入新的问题？

---

### Step 6: 代码审查清单 — 系统性检查

**核心原则**: 修复后必须系统性检查，防止引入新的问题。

**检查清单**:

**数据一致性**:
- [ ] 前端 state、后端内存、数据库/磁盘三者是否一致？
- [ ] 是否有中间状态未被同步？
- [ ] 是否有缓存导致的数据不一致？

**边界条件**:
- [ ] 空文件、大文件、快速输入时是否正常？
- [ ] 网络延迟、超时、断开时是否正常？
- [ ] 并发操作时是否正常？

**副作用**:
- [ ] 修复 A 是否引入了 B 的问题？
- [ ] 是否有遗漏的监听器未清理？
- [ ] 是否有状态未重置？

**代码质量**:
- [ ] 是否移除了调试日志？
- [ ] 是否有死代码或注释掉的代码？
- [ ] 是否有可以复用的逻辑未抽取？

## 常见反模式

| 反模式 | 说明 | 正确做法 |
|--------|------|---------|
| **"我感觉是 XXX 的问题"** | 没有代码或日志支撑 | 用代码/日志验证假设 |
| **"我改一下试试"** | 随机修改代码碰运气 | 先定位根因，再有针对性地修复 |
| **"应该是这里的问题"** | 凭直觉跳过排查步骤 | 按六步法逐步排查 |
| **"修复了就行"** | 不停留在根因层面 | 追问"为什么会这样"直到找到深层根因 |
| **"测试过了"** | 只测试了正常路径 | 测试边界条件和异常情况 |
