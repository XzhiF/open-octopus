# 设计 vs 计划一致性审查

## 概述

当用户提供一份设计文档和一份实现计划，要求审查两者之间的一致性时，使用此技能。核心目标是：**独立判断、找出问题、提供证据**。

不修改设计文档或计划文件。输出只写入 `docs/superpowers/discussion/` 目录。

**输出目录（固定）：** `docs/superpowers/discussion/`
**报告命名（固定）：** `YYYY-MM-DD-<topic>-review.md`
**验证目录（固定）：** `tmp/`（仅当需要最小代码验证时使用）

**开始时宣布：** "正在审查设计文档与实现计划的一致性。"

## 审查流程

### 阶段 1：读取两文档

完整阅读设计文档和实现计划，建立心智模型：
- 设计文档：决策记录、架构、接口定义、数据流
- 实现计划：文件结构、任务拆分、代码片段、测试策略

### 阶段 2：提取可验证的断言

从两份文档中提取所有**可对代码库验证**的断言：

- 文件路径（如 `packages/shared/src/types/workflow.ts:102-112`）
- 函数/类型/接口名称（如 `WorkflowSchema`、`isOctopusWorkflow`）
- 代码行为描述（如 "返回 `{ ref, name }[]`"）
- 依赖关系（如 "import { parseWorkflow } from @octopus/shared"）
- 技术约束（如 "使用 ESM"、"Zod schema"）

### 阶段 3：逐一验证

对每个可验证断言，**读取实际源文件**确认其真实性。不要假设计划或设计中的描述是正确的。

验证方法优先级：
1. **Read** — 读取引用的源文件，比对断言与实际代码
2. **Grep** — 搜索函数/类型定义确认存在与位置
3. **Bash（最小代码验证）** — 仅当需要运行时验证才能确定问题时，在 `tmp/` 下建立最小复现

**最小验证原则**：
- 只在阅读源码仍无法确定结论时才写验证代码
- 验证代码写入 `tmp/<feature>-verify/` 目录
- 验证代码必须是最小独立可运行的，不依赖项目构建
- 完成验证后不需要删除（供用户核查）

### 阶段 4：分类问题

按三类问题整理发现：

| 类别 | 定义 | 严重度 |
|------|------|--------|
| **矛盾** | 设计与计划在同一问题上做出不同假设或选择，无法同时成立 | HIGH/MEDIUM/LOW |
| **设计缺陷** | 设计或计划中的选择本身存在问题（性能、安全性、可维护性、一致性） | HIGH/MEDIUM/LOW |
| **计划遗漏** | 设计中明确要求的点在计划中没有对应任务；或必要的工程实践缺失 | HIGH/MEDIUM/LOW |

严重度标准：
- **HIGH** — 阻塞性问题，必须解决才能继续实现
- **MEDIUM** — 应修复，但不阻塞
- **LOW** — 建议修复，可延后

### 阶段 5：输出审查报告

将分析结果写入 `docs/superpowers/discussion/YYYY-MM-DD-<topic>-review.md`。

## 审查维度

审查时必须覆盖以下维度：

### 1. 数据流一致性
- 输入/输出类型在设计和计划中的定义是否一致？
- API 返回格式是否匹配前端消费方式？
- 数据从源头到消费者的链路是否完整？

### 2. 接口契约
- 函数签名、props、路由参数在两文档中是否一致？
- 新增类型是否在所有引用处正确定义？
- 是否存在重复定义或类型漂移？

### 3. 架构边界
- 各层（shared/server/web-app/cli）的职责划分是否一致？
- 跨层引用路径是否正确？
- 配置/环境变量是否统一管理？

### 4. 技术可行性
- 代码是否符合项目现有技术栈约束（ESM/TypeScript/Zod/Hono 等）？
- 是否存在与现有代码冲突的变更？
- 任务依赖顺序是否合理？（前置任务的输出被后续任务使用时，任务顺序是否正确？）

### 5. 完整性与覆盖
- 设计的每个章节/决策是否在计划中有对应任务？
- 计划中的每个任务是否都有设计的依据？
- 变更影响范围（设计中的变更范围表）是否在计划中全部覆盖？

### 6. 工程质量
- 是否存在重复代码（DRY 违规）？
- 错误处理是否完整？
- 类型安全是否充分？
- 测试覆盖是否合理？

## 关键原则

### 独立性（CRITICAL）
- 不要看 `docs/superpowers/discussion/` 目录中已有的审查报告
- 不要受设计或计划文档中"已批准"、"决策记录"等字眼的影响
- 用自己的眼睛验证每一个可验证的断言
- 即使设计文档写"方案 A：选定"，如果方案 A 在源码中不可行，也要指出

### 证据驱动
- 每个问题必须附具体证据：文件路径 + 行号 + 实际代码片段
- 不能说"可能"或"应该"，必须基于实际读取的源码
- 如果无法确定，明确标注为不确定性并建议进一步调查方向

### 精确但不可修改
- 指出问题时给出精确的位置引用
- 不修改设计文档或计划文件
- 输出只写入 `docs/superpowers/discussion/`

### 不假设
- 不假设"这个 TODO 以后会处理"——如果它是问题，就指出
- 不假设"设计文档写的路径是对的"——验证
- 不假设"这个类型后面会定义"——检查全链路

## 反模式

- 只看文档不看源码做判断
- 问题描述模糊，没有具体行号或代码片段
- 对不确定的断言跳过验证
- 过度谨慎——即使小问题也应该明确记录
- 只关注矛盾而忽略设计缺陷和计划遗漏
- 重复设计和计划已经自检过的内容（除非发现新的具体问题）
