# 工作流自进化系统（Self-Evolving Workflow System）

> 来源：PR #20 设计思路提炼 + 审查反思
> 状态：设计研究，非实现方案

工作流自进化是一种让编排系统**自己监控自己、发现问题、提出改进、执行变更**的能力。不同于传统的"人定义工作流 → 机器执行"模式，自进化系统在工作流运行过程中持续积累度量数据，基于规则和多专家协作识别优化与淘汰机会，最终通过自动化 PR 闭环落地变更。

---

## 核心思想：四层进化架构

```
┌─────────────────────────────────────────────────────────┐
│                    第四层：闭环执行                        │
│         cron 触发 → 分析 → 讨论 → PR → 人类审批           │
├─────────────────────────────────────────────────────────┤
│                  第三层：多专家协作决策                     │
│     architect / security / product 各给意见 → 综合        │
├─────────────────────────────────────────────────────────┤
│                   第二层：自我优化                         │
│   淘汰分析 / 优化建议 / 外部趋势 / 进化提案生成             │
├─────────────────────────────────────────────────────────┤
│                   第一层：自我感知                         │
│      执行统计 / 失败率 / 耗时 / 队列积压 / rate limit       │
└─────────────────────────────────────────────────────────┘
```

### 第一层：自我感知

**核心问题**：大多数工作流系统只知道"跑了没有"，不知道自己"跑得好不好"。

**解决方案**：给每个工作流建立健康度量体系。

| 度量维度 | 指标 | 数据来源 |
|----------|------|----------|
| 使用率 | 执行次数 / 时间窗口 | 执行记录表 |
| 可靠性 | 失败率、连续失败次数 | 执行记录表 |
| 效率 | 平均耗时、P95 耗时 | 执行记录表 |
| 系统健康 | 队列积压深度、执行延迟 | 调度引擎状态 |
| 外部约束 | GitHub API rate limit 剩余 | `gh api rate_limit` |

**设计原则**：度量驱动，不是 LLM 拍脑袋。阈值（失败率 > 50%、耗时 > 600s）可配置可调，不是黑盒。

### 第二层：自我优化

**核心问题**：有了度量数据，如何转化为可执行的改进？

**三条优化路径**：

#### 路径 1：低效工作流优化

WorkflowAnalyzer 基于度量数据识别问题模式：

| 问题模式 | 判定规则 | 优化建议 |
|----------|---------|---------|
| 高失败率 | 失败率 > 50% | 加重试、加错误处理 |
| 长耗时 | 平均耗时 > 600s | 拆分步骤、加并行化 |
| 串行可并行 | 节点间无依赖但顺序执行 | 改为 DAG 并行模式 |
| 低使用率 | 30 天内执行 < 5 次 | 标记为淘汰候选 |

#### 路径 2：工作流淘汰

RetireAnalyzer 管理"工作流的生命终点"：

- **淘汰候选**：使用率低 + 失败率高 + 不在保护名单
- **保护机制**：`retire_protected` 列表防止误删关键工作流
- **生命周期闭环**：发现 → 标记 → 审批 → 归档/删除

这是很多系统缺失的一环 — 只有"创建"没有"淘汰"，工作流越积越多。

#### 路径 3：外部趋势感知

FrontierScraper 不只看内部数据，还看外部：

- GitHub trending repos（新工具、新框架）
- AI papers / 新方法论
- 社区最佳实践

**价值**：系统不只是优化存量，还能引入增量 — "发现新工具 → 建议替换旧方案"。

### 第三层：多专家协作决策

**核心问题**：单一 LLM 做优化决策容易有盲区。

**解决方案**：让多个 AI 专家从不同视角分析同一提案，综合后输出。

```
                    ┌──────────────┐
                    │  进化提案输入  │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ Architect│ │ Security │ │ Product  │
        │ 架构影响  │ │ 安全风险  │ │ 用户价值  │
        └────┬─────┘ └────┬─────┘ └────┬─────┘
             │            │            │
             └────────────┼────────────┘
                          ▼
                    ┌──────────────┐
                    │  Host 综合    │
                    │  加权/投票    │
                    └──────┬───────┘
                           ▼
                    ┌──────────────┐
                    │  综合意见输出  │
                    └──────────────┘
```

**关键设计约束**：
- **专家上限 5 个** — 防止无限讨论，控制成本
- **JSONL 讨论日志** — 完整记录每个专家的意见，可追溯"为什么做了这个决定"
- **共识检测** — 多轮讨论时评估 consensus_score，达标提前退出
- **退化链** — Host 综合失败时 opus → sonnet → 简单拼接，保证总有输出

### 第四层：闭环执行

**核心问题**：分析和讨论的结果如何落地？

**解决方案**：自动化 PR 创建 + 人类审批的半自动闭环。

```
cron 定时触发
    │
    ▼
自我感知（采集度量数据）
    │
    ▼
自我优化（识别问题 + 生成提案）
    │
    ▼
多专家讨论（验证提案可行性）
    │
    ▼
GitHub PR Creator（自动创建 PR）
    │
    ▼
人类 Review + Merge（最终决策权在人）
```

**设计哲学**：AI 做分析和准备工作，人类做最终决策。不是全自动，是**有人类兜底的自动化**。

## 六个进化工作流的分工

| 工作流 | 职责 | 触发方式 | 频率 |
|--------|------|---------|------|
| knowledge-update | 扫描 skills 变更，更新知识库索引 | cron | 每日 |
| workflow-optimize | 分析低效工作流，生成优化 PR | cron | 每周 |
| workflow-retire | 识别冗余工作流，生成淘汰 PR | cron | 每月 |
| frontier-explorer | 发现外部新工具/新趋势 | cron | 每周 |
| swarm-discuss | 多专家讨论进化提案 | 按需 | 提案产生时 |
| meta-evolution | 修改自身进化配置（调整阈值、保护名单） | 按需 | 积累足够数据后 |

**`meta-evolution` 是最有深度的设计** — 让进化系统修改自己的配置。当系统发现"失败率阈值 50% 太高了，很多低效工作流没被识别"，它可以提出 PR 把阈值调到 30%。这是真正的**自指**（self-reference）。

## 度量体系设计

### 工作流健康评分模型

```
HealthScore = Reliability × 0.4 + Efficiency × 0.3 + Utilization × 0.3

Reliability = 1 - failureRate
Efficiency  = 1 - (avgDuration / maxExpectedDuration), capped [0, 1]
Utilization = executionsInWindow / expectedExecutions, capped [0, 1]
```

### 淘汰候选评分模型

```
RetireScore = (1 - utilization) × 0.5 + failureRate × 0.3 + stalenessDays × 0.2

淘汰条件：RetireScore > threshold AND NOT in retire_protected
```

### 低效检测规则

| 规则 | 阈值 | 建议动作 |
|------|------|---------|
| 失败率过高 | > 50% | 加重试 / 检查依赖 |
| 耗时过长 | > 10 分钟 | 拆分 / 并行化 |
| 连续失败 | > 3 次 | 暂停 + 告警 |
| 低使用率 | 30 天 < 5 次 | 评估是否还需要 |

## 关键设计原则

### 1. 度量驱动 > LLM 猜测

先用结构化数据定义"什么是差"，再用规则触发优化。LLM 用于生成人类可读的优化建议和 PR 描述，不用于判断"是否需要优化"。

**为什么**：度量是确定性的、可审计的、可回溯的。LLM 判断是概率性的、不可复现的。

### 2. 淘汰和创建一样重要

大多数系统只管"创建工作流"。自进化系统必须管理完整生命周期：

```
创建 → 运行 → 度量 → 优化 → 淘汰
```

没有淘汰机制的系统会逐渐积累大量"僵尸工作流"，增加认知负担和维护成本。

### 3. 外部感知防止内卷

只看内部度量会导致"局部最优" — 把一个已经很差的系统优化到"不那么差"。外部感知引入"别人在用什么"的视角，可能发现"不需要优化旧工作流，而是用一个新工具替换它"。

### 4. 多视角降低决策风险

单一模型/单一视角做优化决策容易漏掉：
- 架构影响（这个优化会不会引入循环依赖？）
- 安全风险（这个变更会不会暴露敏感数据？）
- 用户价值（这个工作流虽然效率低，但是关键业务依赖它）

多专家讨论不是增加复杂性，是**用多样性对冲盲区**。

### 5. 自指需要护栏

`meta-evolution` 让系统修改自己的配置，这很强大也很危险。护栏：
- 所有配置变更必须通过 PR，人类审批
- 保护名单（`retire_protected`）不可被自动修改
- 配置变更频率限制（不能每次 cron 都改）
- 变更审计日志（谁改了什么，什么时候改的）

## 实现教训（PR #20 审查反思）

### 不要一次交付 6700 行

6 个工作流同时铺开，每个都是 stub。正确做法：**一次做一个工作流，端到端做完再下一个**。

推荐顺序：
1. knowledge-update（最简单，纯本地操作）
2. workflow-optimize（需要度量数据积累）
3. workflow-retire（依赖 optimize 的度量基础设施）
4. frontier-explorer（独立模块，可以后做）
5. swarm-discuss（依赖 proposal 输入）
6. meta-evolution（最复杂，最后做）

### 测试必须在真实代码上跑

PR 的 `discussion-coordinator.test.ts` 用了假角色名，`KNOWN_ROLES` 校验会直接 throw — 说明测试写完**从来没在真实代码上跑过**。"145 tests ALL PASS"是自审报告的数字，实际跑出来 72 个失败。

**教训**：AI 生成的测试代码必须真实执行验证，不能信任"生成者说都过了"。

### 审计报告的可信度

PR 自审报告声称：
- "145 tests ALL PASS" → 实际 72 失败
- "HTTP routes 已修复" → 一半返回 stub
- "安全风险已修复" → 路径遍历和 SQL 注入仍在

**教训**：AI 自审报告的数字需要独立验证。审计者和被审计者不能是同一个 AI。

### 先安全后功能

路径遍历、SQL 注入、零认证这些安全问题应该在第一个 commit 就解决，不是等审查发现后再补。**安全不是 feature，是 baseline**。

## 与现有系统的关系

| 能力 | Octopus 已有 | 自进化补充 |
|------|-------------|-----------|
| 工作流定义 | YAML + 7 种执行器 | — |
| 工作流执行 | WorkflowEngine + DAG | — |
| 执行记录 | SQLite executions 表 | UsageTracker 聚合分析 |
| 多智能体 | SwarmExecutor | DiscussionCoordinator 用于进化决策 |
| 定时触发 | Scheduler + cron | 进化工作流的 cron 配置 |
| 人类审批 | ApprovalExecutor | PR review 作为最终关卡 |

自进化系统**不引入新基础设施**，而是在已有能力之上增加"度量 → 分析 → 决策 → 执行"的闭环层。

## 落地路线图建议

### Phase 1：度量基础（1-2 周）
- UsageTracker 服务（真实聚合查询）
- Dashboard 过滤 tracker 行
- 基础健康评分 API

### Phase 2：分析能力（1-2 周）
- WorkflowAnalyzer（低效检测规则引擎）
- RetireAnalyzer（淘汰候选 + 保护名单）
- 安全加固（SQL allowlist、路径校验、认证）

### Phase 3：闭环执行（2-3 周）
- PRCreator（安全加固后）
- workflow-optimize 工作流端到端跑通
- workflow-retire 工作流端到端跑通

### Phase 4：高级能力（持续迭代）
- FrontierScraper（外部趋势感知）
- DiscussionCoordinator（真实 Agent SDK 集成）
- ProposalGenerator（基于度量数据的结构化提案）
- meta-evolution（自指配置修改）

每个 Phase 独立 PR，独立测试，独立验证。不做 6700 行一次性交付。
