# Loop Engineering（循环工程）

Loop Engineering 是一种最新的 AI 工程范式，其核心是设计一个能自动运行、自我迭代的 AI 工作流系统，将人类从重复的提示和监控中解放出来，转而专注于规则设计和最终判断。

---

## 核心理念与演进

### 核心理念与目标

Loop Engineering 旨在将人与 AI 协作的模式，从传统的一问一答手动驱动，升级为一个能**自主发现任务、分配工作、检查结果并决定下一步**的自动化闭环系统。其目标是优化任务的生命周期管理，实现 AI 的自治运行。

### 在 AI 工程范式中的定位

它是 AI 工程范式演进的**第三代**，建立在 Prompt Engineering（优化单次交互）和 Harness Engineering（为单次任务提供安全环境）之上。三者的核心区别在于：

| 范式 | 关注点 | 人类角色 |
|------|--------|---------|
| **Prompt Engineering** | 怎么问 | 逐轮操作的提问者 |
| **Harness Engineering** | 在哪里安全运行 | 系统的搭建者，但仍需手动触发任务 |
| **Loop Engineering** | 如何让 AI 自主持续运转 | 规则的设计者，系统可按计划或事件自动触发 |

## Loop 的核心架构：六大构建模块

一个完整的 Loop 系统通常由**五个核心功能模块**和一个**跨会话的记忆层**构成，它们共同协作实现自动化闭环。

### 1. 自动化 (Automations)

负责触发整个循环的心跳或定时任务，可以基于固定时间（如 Cron 任务）或特定事件（如代码提交、CI 失败）自动启动，是实现**无人值守**的关键。

### 2. 工作树 (Worktrees)

解决多个 AI 智能体并行工作时的文件冲突问题。它通过为每个智能体创建独立的工作目录，确保它们修改同一代码库的不同副本，互不干扰。

### 3. 技能 (Skills)

将项目特定的知识、规范、架构和最佳实践固化下来，存储为外部文件（如 `SKILL.md`）。AI 智能体在每次执行任务时都会读取，避免每次冷启动都需要重新解释项目背景，实现了**经验的累积和复用**。

### 4. 插件与连接器 (Plugins / Connectors)

赋予 Loop 与真实世界工具交互的能力。通过 MCP 等协议，Loop 可以连接数据库、读取 CI 日志、操作 Issue 跟踪系统（如 Linear）、调用 API 等，从仅能读写文件的小循环升级为能**实际干活的大循环**。

### 5. 子智能体 (Sub-agents)

将执行与验证职责分离的关键设计。通常设置一个**制造者 Agent** 负责生产（如写代码），另一个独立的**检查者 Agent** 负责评审。这种交叉检查机制极大地提高了无人值守时输出结果的可靠性。

### 6. 记忆/状态 (Memory/State)

作为整个循环的脊椎或状态记录仪。它通常是一个存储在磁盘上的 Markdown 文件、Linear 看板或数据库，用于持久化记录任务进度、尝试过的方案、成功与失败的结果。这确保了 Loop 在跨会话运行时能够**记住历史，实现工作的连续性**。

## 适用场景与关键挑战

### 典型适用场景

Loop Engineering 尤其适合**流程稳定、重复发生、且结果有明确验证标准**的任务。例如：

- **软件开发**：自动化代码巡检、依赖升级、测试修复、每日 CI 失败分析与分诊
- **内容运营**：自动进行选题发现、资料整理、初稿生成与事实核查
- **客服与产品**：自动分类客户问题、整理用户反馈与竞品信息、生成日报或简报
- **研究分析**：持续追踪特定领域的新论文、报告与数据，并整合分析

### 主要挑战与边界

#### 成本与验证

Loop 一旦运行，会反复调用 AI 进行试错和验证，可能导致较高的 Token 消耗。AI 输出的完成声明仍需人类最终验证，**不能完全替代人的判断责任**。

#### 理解债务与认知投降

Loop 产出代码的速度可能远超人工理解的速度，导致开发者对项目底层逻辑的**理解债务快速积累**。同时，过度依赖自动化可能诱使开发者放弃思考，直接接受所有结果，即**认知投降**。

#### 适用边界

对于一次性、目标模糊或难以客观验证的任务，Loop Engineering 的投入产出比可能不佳。
